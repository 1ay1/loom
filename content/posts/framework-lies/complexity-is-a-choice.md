Do you know how much software it takes to display "Hello, World" in a browser in 2026?

In 1991, the first web server ran on a NeXT workstation. The HTTP server, the HTML renderer, and the network stack all fit comfortably in 8MB of RAM.

A "Hello, World" web app generated by a modern framework starter template now requires:

- A JavaScript runtime: ~30MB
- A `node_modules` directory: ~150MB
- A Webpack/Vite build artifact: ~50MB
- A Docker base image: ~200MB

**~500MB to render two words on a screen.**

We made the software 62,500 times larger to do the same thing. We call this progress.

## The Dependency Black Hole

Open any project generated by a popular JavaScript framework. Run `npm list`. Count the direct dependencies. Then run `npm list` with full depth. Count again.

The second number is not the first number plus a few. It's the first number times fifty.

You wanted a router. The router depends on a path-matching library. The path-matching library depends on a string escaping utility. The string escaping utility depends on a regular expression helper. The regular expression helper was last updated in 2019, has three open security advisories, and gets 40 million weekly downloads because it's deep in the tree of something everyone uses.

Every one of those packages is:
- A potential security vulnerability you didn't choose and can't easily remove
- A potential breaking change on its next semver bump
- Code written by a stranger whose maintenance priorities may not align with yours
- Something you have to audit, update, and worry about forever

Loom — the server running this blog — has zero external dependencies beyond `zlib`. The compiled binary is **800KB**. Not 800MB. 800KB. The container image, if you want one, is under 2MB.

That's not a flex. That's what software looks like when you only add things you actually need.

## The Toolchain That Ate Itself

Here is the history of JavaScript build tooling, compressed:

1. **JavaScript was too slow** at startup → bundlers (Browserify, Webpack)
2. **Webpack was slow to configure** → zero-config bundlers (Parcel)
3. **Parcel was slow to build** → faster bundlers rewritten in Go/Rust (esbuild, SWC)
4. **Raw esbuild was tedious to manage** → dev server wrappers (Vite)
5. **Vite alone wasn't enough** → meta-frameworks wrapping Vite (Next.js, Nuxt, SvelteKit)
6. **Meta-frameworks were complex to configure** → starter templates pre-configuring the meta-frameworks
7. **Starter templates diverged too much** → framework-specific CLIs enforcing the templates

We spent a decade solving problems created by the previous decade's solutions. Each layer exists because the layer below it was broken or inadequate. The current state of the art is seven abstraction layers on top of a language that runs in a virtual machine inside a container on a server that's running Linux inside a hypervisor.

At each step, someone wrote a blog post about how this new layer was going to "simplify everything."

## Resume-Driven Development

There's a name for the practice of choosing technologies because they look good on a CV rather than because they solve the problem at hand.

**Resume-Driven Development.**

It is not a fringe practice. It describes a meaningful percentage of all architectural decisions made at small-to-medium companies.

The engineer who proposes microservices gets to put "microservice architecture design" on their resume. The engineer who implements Kubernetes gets "Kubernetes" on theirs. The engineer who builds a simple, boring monolith that works perfectly for five years gets to write... "maintained backend system."

The incentive structure rewards complexity. Complexity reads as seniority. Simplicity reads as naïveté — even when the simple system outperforms the complex one by every metric that matters.

The people funding the infrastructure often don't have the technical context to push back. The result: small teams running infrastructure appropriate for organizations ten times their size, paying cloud bills that are ten times what they need to be, while the engineers are fully employed just keeping the complexity from collapsing.

## The Hidden Cost of "Modern"

"Modern" has become a virtue signal detached from technical meaning.

A modern stack is not better than an older one by virtue of being newer. A modern stack is better if it solves more problems than it introduces. Most modern stacks do not clear this bar.

When you choose a framework, you're not just choosing how to write the feature today. You're choosing:

- How hard it will be to debug in 18 months when you've forgotten the conventions
- How long it takes to onboard the next engineer
- What your upgrade path looks like when the framework ships a breaking major version
- How much of your engineering time is "building product" vs. "maintaining build pipeline"

These costs are invisible on day one. By month twelve, they're the dominant cost.

## Simplicity Is an Achievement, Not a Shortcut

The hardest thing to do in software is say no.

No, we don't need this dependency for something we can write in 40 lines.
No, we don't need a distributed queue for 3,000 daily users.
No, we don't need Kubernetes for a two-container application.
No, we don't need a meta-framework for a site that serves HTML.

Saying no is hard because complexity provides cover. If the system is complex, failures are "infrastructure issues," not code problems. If the system is complex, you're a "platform engineer." If the system is complex, removing things feels risky because nobody fully understands it anymore.

Complexity is job security for individuals. It is a slow drain for the organization paying for it.

But simplicity is also: a $6/month server instead of a $600/month cluster. A 10-second build instead of a 10-minute one. A new engineer productive in two days instead of two weeks. A bug found in 5 minutes instead of 3 hours. A system that runs for three years without anyone touching it, because there's nothing to go wrong.

Loom is 800KB. No dependencies. 10-second build. Sub-millisecond response times on hardware that costs less than a coffee per month. Not because it does something magic — because it refuses to do thousands of unnecessary things.

That's the other end of the spectrum. It's available to anyone willing to make different choices.
